- **Frontmatter schema** (`pkg/agent/llmtools/skill/types.go:70-79`): `name`, `description`, optional
  `license`, optional `compatibility`, optional free-form `metadata` map. Spec ref: agentskills.io
  (`types.go:69`).
- ⚠️ **`allowed-tools` is NOT supported**: literally `// TODO: support allowed-tools` (`types.go:78`).
  Confirms Plan A/B: per-skill tool restriction must live in the **harness/agent config**, not skill metadata.
- ⚠️ **`compatibility` is parsed and length-checked (≤500 chars) but NOT behaviorally enforced**
  (`types.go:74,92-94`). A Laguna library can *declare* model compatibility, but `pool` won't act on it.
  Model targeting/routing must be harness-side.
- **Name rules** (`types.go:39,46-61`): `^[a-z0-9]([a-z0-9-]{0,62}[a-z0-9])?$`, ≤64 chars, no leading/trailing
  or consecutive hyphens. **Description required, ≤1024 chars** (`types.go:86-91`). *(All of Plan A's proposed
  names (`laguna-task-contract`, `ci-log-reducer`, `repo-map`, `stack-trace-router`, `single-file-patch`,
  `regression-test-generator`) are valid.)*
- **In-repo authoring precedent already exists:** `pool` ships a default **`skill-creator`** skill
  (`pkg/agent/llmtools/skill/resources/default_skills/skill-creator/SKILL.md`) with a `references/` subdir and
  `LICENSE.txt`, plus `metadata.version: "0.2.0"`. The other two defaults (`configure-sandbox`,
  `pool-product-reference`) show the house style (concise `SKILL.md`, `metadata.version` semver, description
  written as explicit trigger phrases). **Workstream A should align with these conventions** rather than invent new ones.
