- **Discovery locations**: workspace `.poolside/skills` and `.agents/skills` (`local_registry.go:33`);
  user-global `~/.config/poolside/skills` (`types.go:102`); remote repositories (`types.go:107`).
- **Composite registry** = local + optional remote; **dedupe by name, local beats remote**
  (`composite_registry.go:15-34`).
- **`RegistryConfig` fields** (`types.go:11-21`): `WorkingDirs`, `SkipDefaultSkillInstall`, `HTTPClient`,
  `RemoteRepositories`.
- ⚠️ **Default skills auto-install on registry init unless skipped**:
  `if !config.SkipDefaultSkillInstall { installDefaultUserGlobalSkills() }` (`local_registry.go:41-43`).
  Defaults are **`go:embed`-bundled** (`default_skills.go:22`), **versioned by `metadata.version` + semver**,
  and **rewritten/upgraded to user-global whenever the embedded version is newer** (`default_skills.go:80,86-98`).
  Shipped defaults today: **`configure-sandbox`, `pool-product-reference`, `skill-creator`**.
  Planning consequence: a normal `pool exec` run mutates the user-global skills dir and merges those skills into availability. `SkipDefaultSkillInstall` only prevents installing/upgrading defaults; it does not stop already-present user-global skills from being scanned. A clean "no-skill" arm needs stronger isolation: skill tool disabled, an isolated/empty skills root + HOME, or a workspace-only / empty-allow-list registry.
- ⚠️ **Skills are ENABLED BY DEFAULT in `pool exec`**: `types.SkillToolName` is in the default tool set
  (`config/defaults.go:70`), and the registry is constructed whenever that tool is enabled
  (`configbuilder/builder.go:413`). *(Plan B implies skills work in exec but doesn't state they're on by
  default; they are.)* These are **two distinct risks** to control separately: (i) the skill *tool being available at all* can change model behavior, vs (ii) *discovered skills contaminating* output. "Tool disabled" and "tool enabled with zero available skills" are different experimental controls.
- ✅ **Enabler (the flip side):** because the skill tool is on by default and local `.poolside/skills` / `.agents/skills` discovery already works, **workspace skills can be authored and run on today's `pool exec` with no forge change**. That is the foundation of "buildable now" (§3.1). The default-install + user-global behavior is the *contamination* risk to *control* for clean comparisons, not a blocker to starting.
