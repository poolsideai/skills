ps poolside skills — UX review round 1 of 2 · audit + IA + 3 directions

One substrate, three apps, zero paths: a UX audit of the skills workbench

The data model is right — everything reduces to a trajectory record, and skill lift is measurable. But the UI fragments one mental model across three surfaces, buries both creation flows, and never offers the system's most basic verb: run a skill against a prompt, right now.

audited: catalog :4319 workbench /workflows.html review app :8901 ui/lib.ts · server.ts · bench.ts
TL;DR — FIVE CALLS
  1. Collapse to one app, four nouns. Skills · Workflows · Runs · Review, one theme, one nav. The catalog stays a public export of the same data, not a sibling app.
  2. One authoring surface, chat-driven. A single composer authors skills and workflows, iterates by rewriting, supports $skill mentions, and hands you node-by-node manual control after generation.
  3. Add the missing primitive: the playground run. "Try this skill with a prompt" is the zero-to-first-success path and the cheapest trajectory producer. It doesn't exist today.
  4. One runs feed with facets, not three look-alike lists. Workflow runs, eval arm-runs, node evals, and playground runs are one record shape — show them as one feed, faceted by skill / workflow / arm / verdict, with the skill scorecard as its pivot.
  5. Make review a mode, not a second app. Deep-link any run row into a full-screen annotation mode; labels land in the skill's improvement queue — the seed of the DSPy/GEPA loop.
01

The mental model, stated once

The confusion isn't that the concepts are complicated — it's that the UI never states them. Here is the whole domain in one sentence: a skill is a graded contract; a workflow is a graph of agent nodes that may install skills; every execution is a trajectory record; an eval is a graded record; review turns grades into improvements.

THE NOUNS & HOW THEY CONNECT
SKILL
A contract: output schema + executable validator + eval cases. Exists to lift model outcomes.

NODE ⊂ WORKFLOW
One agent turn: a prompt + a model + 0…n skills. A workflow is a Smithers graph of these.

Any execution — workflow, eval arm, node trial, playground. All one record shape.

EVAL → LABEL
Validator grade (mechanical) + human label (review). Evidence the skill helped — or how it failed.
↩ The loop: failing grades and labels flow back into the skill — prompt tweaks, contract changes, new eval cases. That loop is the product (and the on-ramp to DSPy/GEPA-style automation later). Every screen should make the user's position in this loop obvious.

Four producers emit the same record. Today each producer gets its own list with its own styling and its own scroll position. They should be one feed — and the aggregated vs. disaggregated question (skill ran top-level vs. skill ran as a node inside a workflow) becomes a facet, not a separate section.

02

Audit — what's broken and why

Eight findings, ordered by how much they block the core loop. Your own words are the acceptance test: "I built the thing and I don't know how to just run a skill I already have with a prompt."

HIGH F1 · cross-cutting

Three apps carry one mental model

Observed. Catalog (static, :4319), workbench (same port, different page), review app (:8901, entirely different light theme). Nav labels drift: the catalog's nav says Workflows, the workbench's says Workbench; "Evals" is an in-page anchor on one surface and a separate origin on another. The review app has no link back at all.
Why it hurts. Users must understand your deployment topology (two ports, three documents) to navigate a single conceptual space. Every hop resets visual language, so nothing learned on one surface transfers. This is the root of "I can't keep the mental models clean" — the models are fine; the seams are arbitrary.
HIGH F2 · authoring

Creation — the whole point — is buried and one-shot

Observed. Skill authoring is a collapsed <details> labeled "Author a new skill" at the bottom of the third sidebar panel. The workflow composer is a 5-row textarea in a ~280px column. Both are fire-and-forget: one generate click, one hidden repair round, no rendered output to inspect, no conversation, no per-node tweaking afterward. Everything downstream (runs, evals, the loop) depends on these two flows.
Why it hurts. The system's strongest capability — model-assisted authoring with mechanical verification (smithers graph, check_skill_structure.py) — reads as an afterthought. And because generation is one-shot, the natural behavior ("that's close, now change node 2's prompt") has no home. Authoring should be a conversation that ends in an inspectable, editable artifact.
HIGH F3 · missing primitive

"Run a skill with a prompt" does not exist

Observed. A skill can only be exercised through an eval suite (heavyweight, case-bound) or inside a workflow (requires authoring a graph first). There is no "try it now" anywhere — not on the workbench skills list, not on the catalog detail page (whose "Use this skill" panel offers View source and Copy install command).
Why it hurts. This is the zero-to-first-success path for every new user (including you). It's also the cheapest trajectory producer for the eval flywheel: an interesting playground run should be promotable to an eval case in one click. The substrate already supports it — it's one `pool exec` with a skill installed.
HIGH F4 · runs & evals

Three look-alike run lists; the answer to "did the skill help?" is fine print

Observed. Workflow runs, Evals, and Node evals render as three stacked sections of nearly identical pill-rows; once you've scrolled, the only disambiguator (the heading) is off-screen. There's no grouping or filtering by skill or workflow — the exact pivots you asked for. Skill lift, the headline metric of the entire project, appears as sidebar fine print: "evals — with skill: 0/1 (avg 0.857) · without: 1/1".
Why it hurts. The taxonomy question you raised — top-level skill runs vs. skill-as-node-in-workflow runs, aggregated vs. disaggregated — is unanswerable in this layout because the dimension (skill) is spread across three sections instead of being a facet of one feed. The user is doing the JOIN in their head.
MED F5 · workflow detail

The graph is a poster, not a surface

Observed. The projected graph shows node names and truncated descriptions, but nodes aren't clickable. A node's actual substance — its prompt, its model, its installed skills, its latest grade — is invisible and uneditable. Yet the node is precisely where skills and workflows meet, and per Smithers, each node is a prompt + model + skills.
Why it hurts. "Skills feel disconnected from workflows" is this finding. The connection exists in the data (PoolAgent { skill }, pool captures, node-eval grades) but has no pixels. A node inspector — model select, prompt editor with $skill typeahead, grade chips — is the single highest-leverage missing component.
MED F6 · review

Review is a dead end, and its plumbing is exposed as UI

Observed. "Sync workbench → review" makes the user run an ETL step by hand. "review traces ↗" opens trace 1-of-32 in a foreign app — not the failing run you were just looking at. Labels auto-save to labels.json and… stay there. Nothing routes a Fail label + note back to the skill it indicts.
Why it hurts. You said evals exist for skill improvement. The improvement signal currently terminates in a JSON file. The review UI itself is genuinely good (keyboard-first, gold column, trajectory tree) — it's the connective tissue that's missing: deep-link in from any run row, route labels out to the skill's improvement queue.
MED F7 · layout

One endless column; the sidebar does three jobs

Observed. The workbench stacks graph → runs → run detail → evals → node evals in a single ~5-screen scroll. The sidebar interleaves navigation (workflow list, skills list), creation (two generator forms), and status — three different jobs in one 280px column. Failing-check details dump full-width strings into list rows.
Why it hurts. Position in the page becomes the only information scent, and it's unreliable ("Everything is one endless scroll"). Each job the sidebar does undermines the others: lists get cramped to make room for forms; forms feel minor because they're styled like list footers.
LOW F8 · catalog

Catalog truth drifts from workbench truth

Observed. The skill detail page hand-maintains "Eval status" numbers (and currently renders a stack of duplicate/skeleton Eval status panels). The workbench computes real per-arm scorecards from run manifests. Two sources of the same fact.
Why it hurts. Minor now; corrosive once open-sourced — the public page is the one that will be wrong. Fix falls out of the IA: the catalog becomes a build-time export of the same scorecard data, not a parallel authoring effort.

05

Recommendation

A as the backbone, C as the Workflows section, B's scorecard as the Runs home. These aren't really competing apps — they're competing answers to "what is home." A's entity-first nav is the only one that stays legible for open-source newcomers and gives every noun a stable address. Then the two genuinely novel surfaces slot in where they belong: the Canvas (C) is the workflow detail page, and the Loop scorecard (B) is the Runs landing view. One composer (⌘K) serves all of it.

If you'd rather commit to a pure direction, pick C — it attacks your two worst pains (creation, skill↔workflow disconnect) most directly. But say the word and round 2 is hi-fi screens of the hybrid: Skills page with Try-it panel, Workflow canvas with node inspector + chat, unified Runs feed + scorecard, and Review as a themed full-screen mode.
